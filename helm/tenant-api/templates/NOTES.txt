SECURITY (GHSA-3g2h-rf85-5rrv) — internal port 8080 BYPASSES oauth2-proxy and
  blindly trusts X-Forwarded-Groups / X-Forwarded-Email. Anything that can reach
  8080 can assert ANY identity (incl. platform-admins -> fleet admin write +
  federation token minting + git PR-write). The pod-scoped NetworkPolicy (four
  admitted clients: Prometheus + threshold-govern + recipe-preview + da-portal)
  is the ONLY control on that port — this is why networkPolicy.enabled=false is a
  hard failure.

HUMAN SOCKET (ADR-027 D2-B) — ENABLED. The oauth2-proxy sidecar forwards human
  traffic to tenant-api over a pod-internal Unix socket ({{ .Values.humanSocket.path }}),

  older image crash-loops on the unknown flag. Do NOT flip this via
  `helm upgrade --reuse-values` — pass humanSocket values explicitly.

NetworkPolicy only works if your CNI ENFORCES it. Flannel (base) and some managed
  "simple" CNIs SILENTLY IGNORE NetworkPolicy: the API server accepts it and this
  install succeeds, but port 8080 is left open to the whole cluster (security
  theater). Use a NetworkPolicy-aware CNI (Calico / Cilium / Antrea) and verify by
  sending a packet that SHOULD be denied — do not trust a successful install. The
  CNI-independent root-cause fix is L7 identity verification (roadmap #5); this
  policy is L4 defense-in-depth.

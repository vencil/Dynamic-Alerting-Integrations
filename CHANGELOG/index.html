    Security¶

    • Grafana 12.4.1 → 12.4.2（v2.8.0, #98 of umbrella #100 Q2 2026 CVE audit） — bump k8s/03-monitoring/deployment-grafana.yaml。audit prerequisite confirmed：configmap-grafana.yaml 唯一 data source 是 Prometheus，無 PostgreSQL，故 12.4.1 帶的 pgx/v5 CVE-2026-33816（CRITICAL memory-safety, fixed in 5.9.0）在我們部署是 dead-code。bump 主要修 12.4.1 → 12.4.2 自身的 Go binary CVE（含 grpc CVE-2026-33186 transitive）。Trivy 0.70.0 audit：Go binary 部分 8/2 → 7/1（pgx 那條 CRITICAL 即使可達也已被 12.4.2 fix）。殘留 11 HIGH + 2 CRITICAL Alpine OS layer（OpenSSL/libssl3/musl/zlib）upstream Grafana 還沒 rebase Alpine 3.22+，CRITICAL CVE-2026-31789 是 32-bit only（amd64 N/A），HIGH OpenSSL CMS path 在 Grafana 不啟用 — 留 issue 追 upstream rebase。詳見 #100 audit 摘要。

    • Prometheus + Alertmanager stack bump（v2.8.0, #96 + #97 of umbrella #100 Q2 2026 CVE audit） — bump 4 處 image refs 對齊 prom/ monitoring stack：k8s/03-monitoring/deployment-prometheus.yaml v3.10.0 → v3.11.2、k8s/03-monitoring/deployment-alertmanager.yaml v0.31.1 → v0.32.0、tests/e2e-bench/docker-compose.yml 的 prometheus v2.55.0 → v3.11.2（Prometheus 2.x 已 EOL 2024-12-03）+ alertmanager v0.27.0 → v0.32.0。Trivy 0.70.0 audit 2026-04-26：清掉 grpc CVE-2026-33186（CRITICAL memory-safety）+ Go stdlib 1.25.7 / 1.26.0 → 1.26.2 升級涵蓋的全部 CVE-2026-32280/281/283 等 + e2e-bench 那組 v2.55.0 帶的 11 個 stdlib CVE（Go 1.23.2 → 1.26.2）+ x/crypto CVE-2024-45337（CRITICAL SSH misuse, fixed in v0.31.0）。Prometheus k8s 9 HIGH + 1 CRITICAL → 3 HIGH + 0 CRITICAL（殘留 jsonparser/moby authz/otel BSD-only，皆不可達）；Alertmanager k8s 6 HIGH + 1 CRITICAL → 1 HIGH + 0 CRITICAL（殘留 otel-go kenv path 是 BSD-only）。e2e-bench compose 改 v3.x 後 18 HIGH + 3 CRITICAL → 3 HIGH + 0 CRITICAL（同 k8s 殘留）。注意*：Prometheus 2.x → 3.x 是 major bump，但本 PR 用到的 flags（--config.file、--web.enable-lifecycle、--storage.tsdb.retention.time）都仍向下相容；現有 prometheus.yml / alert-rules.yml 格式無 breaking change。詳見 #100 audit 摘要。

    • PR-1 C1 — Rate limit middleware (components/tenant-api/internal/handler/middleware.go)：homegrown sliding window 80 LoC（不引入第三方依賴 httprate / x/time/rate）。Per-caller 100 req/min default；逾限回 429 + Retry-After + JSON {error, code: "RATE_LIMITED", retry_after_s}。Caller identity 優先序 X-Forwarded-Email → X-Real-IP → RemoteAddr IP（與 rbac.Middleware 一致）。/health//ready//metrics 預設豁免（kube-probe 不燒 budget）。TA_RATE_LIMIT_PER_MIN=0 完全停用；非數值 fallback 預設並 startup log。allow() 用 caller-supplied now 不用 wall-clock — 確保 deterministic 測試（self-review 抓到原 implementation 用 time.Until 在測試模擬時鐘下會產生 retry-after = 1s 而非 30s 的 bug）
    • PR-1 C3 — RequestIDResponse middleware：chi middleware.RequestID 只把 ID 注入 context 不回應；新 middleware mount 在 RequestID 之後，把 middleware.GetReqID(ctx) 寫進 X-Request-ID 回應 header。Customer 端的 correlation ID round-trip 也覆蓋；context 缺值時 defensive no-op 不 crash
    • PR-1 tests (middleware_test.go)：15 cases — RequestIDResponse 三場景 (echo / round-trip / no-op-on-empty) + RateLimit 七場景 (under-cap / at-cap-429 / per-caller isolation / skip-paths / disabled-when-zero / fallback-to-IP / sliding-window-eviction) + 5 cases for RateLimitConfigFromEnv (default / positive / zero / malformed / negative)
    • PR-2 C2 — Tenant-scoped authz on 4 endpoints (components/tenant-api/internal/handler/authz.go + edits to group.go / task.go / pr.go)：(a) PutGroup / DeleteGroup 必須對每個 member tenant 持 PermWrite；缺者全列入 403 訊息（不是 first-only — 同 PR ergonomics 原則 with PR-2 的 forbidden-tenant 列表）；(b) GetTask 過濾 Results[] 為 caller 可讀子集，零可讀子集回 403（task 存在但 caller 無權 — 不是 404）；(c) ListPRs bulk mode 過濾不可讀 tenant；?tenant=<id> mode 不可讀回空列表而非 403（避免 tenant-existence oracle）。Views 不在範圍 — Filters 是 arbitrary metadata 不是 tenant ID list，無對應 RBAC 對象
    • PR-2 tests (authz_test.go)：14 cases — tenantsLackingPermission helper 7 cases (empty / open-mode-read / open-mode-write-rejects-all / read-but-not-write / partial-access / dedup / skip-empty) + PutGroup / DeleteGroup 3 forbidden-member cases + GetTask 2 (filtered / 403-on-zero-access) + ListPRs 2 (bulk-filter / forbidden-?tenant-empty-not-403)

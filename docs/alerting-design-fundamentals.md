# 告警該響之前：從資源飽和到症狀導向的告警設計

> **Language / 語言：** **中文 (Current)** | [English](./alerting-design-fundamentals.en.md)

> **受眾**：剛開始建告警體系的團隊——tenant 團隊與 domain 專家都適用。不需要 SRE 背景。
> **讀完帶走**：一條「何時該吵醒人」的判準、一張「訊號分層」對照、一份自檢 checklist。

## 0. 先問一個問題

你的告警清單裡，有幾條長這樣：`CPU > 90% → critical`？

這篇文章想說服你：這條告警**大多數時候是錯的**——不是閾值錯，是**告警的對象**錯了。錯的告警清單會同時做到兩件壞事：真出事時沒響（漏報）、沒出事時狂響（誤報→告警疲勞→真的響了也沒人理）。要修好它，不用先學任何工具，只要換一個問法：

> **別問「機器怎麼了」，先問「使用者感覺到了什麼」。**

## 1. 症狀 vs 原因：page 的判準

告警訊號分兩種：

- **症狀（symptom）**：使用者可以直接感受到的——頁面變慢、請求失敗、資料是舊的。
- **原因（cause）**：工程師診斷時才看的——CPU、記憶體、連線數、佇列深度。

經典準則是**對症狀告警、少對原因告警**（出處是 Rob Ewaschuk 的 [My Philosophy on Alerting](https://docs.google.com/document/d/199PqyG3UsyXlwieHaqbGiWVa8eMWi8zzAn0YfcApr8Q) 與 [Google SRE Book 第 6 章](https://sre.google/sre-book/monitoring-distributed-systems/)，值得完整讀），理由很機械：一個症狀可能來自幾十種原因（頁面慢：可能是 DB、可能是 GC、可能是網路），一個原因也未必造成症狀（CPU 100% 的機器可能正認真做完所有工作）。對原因告警，等於同時買下漏報與誤報。

要不要**吵醒人（page）**，判準是三個條件同時成立：

1. **緊急**——等到天亮處理就會惡化；
2. **使用者可感**——是症狀，不是內部指標的抖動；
3. **需要人類判斷**——有明確自動化解法的事，該交給自動化（見系列第三篇），不該吵醒人。

三個條件缺任何一個，就降級：不緊急的進 ticket，非症狀的進儀表板，可自動化的進自動化。

這三條之外還有一個隱含前提：響了，得有人**能做點什麼**（actionable）。這個詞再往下游走一格——那個「做點什麼」本身及不及格——就是系列第三篇的全部主題。

## 2. 四大黃金訊號的不對稱：同表不同命

監控教材常給四大黃金訊號：延遲（latency）、流量（traffic）、錯誤（errors）、飽和度（saturation）。很多團隊把四個都接上告警、都設 critical——入門文獻確實說過「四訊號都該量測、出問題就告警」，但那是**底線覆蓋，不是設計目標**：四個訊號的**用途不同**：

| 訊號 | 本質 | 正確去處 |
|---|---|---|
| 延遲、錯誤 | **症狀**——使用者可感 | 告警主力，嚴重時 page |
| 流量 | **脈絡**——解讀其他訊號用 | 儀表板、容量規劃 |
| 飽和度 | **領先指標**——「快要出事」不是「出事了」 | warning／ticket／容量規劃 |

飽和度值得知道，但**多數時候不值得吵醒人**——除非它已經反映成症狀。「CPU > 90% → critical」的問題就在這：它把領先指標當成了症狀。

## 3. 「CPU 滿了」到底該不該叫？

看兩台 CPU 都在 100% 的機器：

- **線上 API 伺服器**：CPU 打滿→請求排隊→延遲飆高→使用者受害。**但正確的告警對象仍是延遲與錯誤率**（症狀），CPU 只是你診斷時會看的原因。
- **跑排程批次的工作機**：CPU 定期打滿是**它在認真工作**。使用者在乎的只有一件事：**工作在期限前完成了沒**。正確的告警是「批次結果的新鮮度」或「完成期限」，不是 CPU。

第二台機器要的告警，文獻裡有現成的類型名——**新鮮度（freshness）**，不用自創。把「使用者感受」量化成可監控的指標，行話叫 **SLI**（Service Level Indicator，服務水準指標）；不同型態的服務，該問使用者的問題不同：

| 服務型態 | 該問使用者什麼 | SLI 類型 |
|---|---|---|
| 請求驅動（API、網站） | 請求成功了嗎？夠快嗎？ | 可用性、延遲、正確性 |
| 管線／批次（ETL、報表、排程工作） | 資料夠新嗎？期限內完成了嗎？ | 新鮮度（freshness）、覆蓋率 |
| 儲存系統 | 存得進、讀得出嗎？ | 可用性、耐久性 |

把你的每個系統對進這張表，告警清單的骨架就出來了——而且你會發現 CPU 不在任何一格裡。

## 4. 反例解剖：把飽和訊號綁上破壞性動作

一個常見的設計模式：某個有狀態服務的連線池打滿時，自動觸發故障轉移、把備援節點扶正。設計動機聽起來合理——「防止連線一直堆積」。但上線後可預期的結局，是它比不動作更糟。三個錯位一次看：

1. **把飽和當故障**。連線池打滿是*負載*問題，不是*節點壞了*。扶正備援沒有移走任何負載——湧入的請求只是轉向，新節點很快同樣打滿。
2. **持續為真的條件 × 一次性的破壞動作**。「連線滿」在負載退去前會一直成立；動作跑完、條件立刻再成立→再觸發→**連環扶正**，整組服務開始翻滾。
3. **摧毀了本來能完成的工作**。原本卡在池裡的連線，很多正在慢慢做完自己的事；每次切換把它們全數清掉——症狀處理設計反而放大了 impact。

正確的拆法，是讓每個訊號各回各位：

- **症狀**（請求延遲、錯誤率）→ page 人來判斷；
- **飽和**（連線池使用率）→ warning + 容量規劃，並在應用層做背壓（backpressure——池滿時讓上游排隊、放慢或拒絕，而不是無上限堆積）——那才是「防堆積」的對症解；
- **故障轉移**保留給真正的故障（節點健康檢查失敗）——那時系統才有一個清楚的「正確樣子」（存在一個健康的主節點）可以自動修回去。

「什麼樣的條件才能安全地掛上自動化動作」是一個值得單獨一整篇的題目——見系列第三篇[《Actionable 之後》](alerting-best-practices.md)，那裡有完整的判準與護欄。

## 5. 角色分工：訊號語意是 domain 專家的事

上面的每個判斷——「這個領域裡什麼算症狀」「飽和到多少算領先警訊」「哪個指標對應使用者的哪個感受」——**都是領域知識**，不該由每個 tenant 團隊各自從零發明，也不該由平台團隊一刀切。健康的分工是：

- **Domain 專家**定義訊號語意：哪些指標是這個領域的症狀、嚴重度（severity）怎麼對應、飽和的合理水位帶。
- **Tenant 團隊**只調自己的目標值：我的服務容忍多少延遲、我的維護窗在何時。

兩邊都需要本文的概念——domain 專家用它設計訊號，tenant 用它理解自己在調什麼。

## 6. 本平台把哪些做成了預設

正文到此是通用準則。下表是本平台的對照（標記沿用系列慣例）：✅ 強制（代碼路徑強制、不可繞過）／⚙️ 預設（預設提供、可配置覆寫）／📖 準則（平台範圍外）。

| 準則 | 本平台 | 機制 |
|---|---|---|
| 症狀導向的告警設計 | ⚙️ | 內建 rule pack（平台按領域打包的告警規則）的設計取向；租戶自訂告警的語意平台不審，適用本文自檢 |
| 訊號語意由 domain 定義、tenant 只調目標 | ⚙️ | rule pack（domain 語意）＋ tenant YAML（閾值）的[配置分層](design/config-driven.md) |
| 批次時段不誤報 | ⚙️ | [排程式閾值](design/config-driven.md)——可配置時間窗（如夜間批次）自動切換較寬閾值，「夜間 batch 假告警」的對症解 |
| 嚴重度分層、只收最高 | ⚙️ | Severity Dedup（預設啟用，租戶可顯式關閉） |
| 格式壞／高基數規則不入生產 | ✅ | Schema 驗證 + cardinality guard（擋格式與基數——標籤組合數爆炸會拖垮監控系統本身；不擋語意壞規則） |
| 飽和訊號降級為 warning／容量規劃 | 📖 | 設計選擇，本文準則；平台的多層嚴重度讓你把它配成 warning |

**想看拆法的現成示範？** 平台內建 rule pack 本身就是照本文原則寫的，可以直接當範本讀（完整目錄見 [Alert 速查](rule-packs/ALERT-REFERENCE.md)）：

- **飽和類告警的基本層一律 warning、critical 是顯式升級**：連線數告警（如 `NginxHighConnections` → `NginxHighConnectionsCritical`）的 critical 層只在你另外設定 `_critical` 後綴閾值後才存在——「這個飽和對我是不是嚴重」是你做的決定，不是預設。
- **升級靠訊號互證——但看清它在層次裡的位置**：`MariaDBSystemBottleneck` 要「連線飽和 **且** 執行緒併發飽和」同時成立 60 秒才升 critical。誠實說：兩個飽和互證，仍然是「組合型的原因」、不是症狀——它是**症狀指標拿不到時的妥協啟發式**（exporter 看不到你的端到端延遲時，「雙飽和同時成立」是最接近「工作正在堵死」的替身）。所以 pack 同時提供了更接近症狀的 `MariaDBHighSlowQueries`（慢查詢率——「變慢」的直接投影），複合告警是它的補充、不是替代。自己設計時的順位：**找得到症狀，就對症狀告警；找不到，才用多訊號互證墊高原因告警的可信度**——別學成「把多個硬體指標綁在一起就能設 critical」。
- **告警文案把注意力導向容量，不是動作**：飽和類告警的說明文字寫的是「檢查連線集區大小」「容量擴增」——訊號的語意設計，連 annotation 都是一部分。

自己寫自訂告警時，照這三個模式抄。

## 7. 自檢 checklist

拿你現有的告警清單，逐條問：

1. 這條告警響的時候，**使用者感覺到了什麼**？答不出來→它是原因告警，考慮降級為儀表板。
2. 它同時滿足**緊急＋使用者可感＋需要人類判斷**嗎？缺一個就降級（ticket／儀表板／自動化）。
3. 清單裡的**飽和類指標**（CPU、記憶體、連線數、磁碟、佇列深度）有幾條掛著 critical？每一條都問：它反映成症狀了嗎？沒有→降 warning、進容量規劃。
4. **批次／排程工作機**的告警，是問「資源」還是問「期限內完成了沒」？改問後者。
5. 有沒有**自動化動作掛在飽和訊號上**？有→先拆下來，讀系列第三篇再決定掛回去的條件。
6. 訊號語意是 **domain 專家定義**的，還是每個團隊自己猜的？
7. 這份 checklist 不是一次性的：**每月例行跑一遍**——30 天內被忽視兩次以上的告警，降級或刪除。告警清單是活的；做這件事不必追究「當初誰設的」——吵鬧的告警是系統設計的缺陷，不是某個工程師的能力問題。

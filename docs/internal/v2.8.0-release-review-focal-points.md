## 2. 高風險量測 surface（reviewer 應該不信任直到自己驗）

### 2.1 Benchmark numbers

| 數字 | 出處 | 該懷疑什麼 |
|---|---|---|
| **Cold load 112 ms @ 1000 tenants** | `docs/benchmarks.md` §3 | (a) 用 synthetic fixture（Zipf+power-law），不是 customer corpus；customer 形狀可能不同。(b) 量測在 Dev Container 內，不是真實 K8s pod。(c) [issue #142 customer-anon corpus calibration](https://github.com/vencil/Dynamic-Alerting-Integrations/issues/142) 已 codify 為「客戶導入時若需 SLA contract anchor」的 escape hatch — 不是 v2.8.0 release 的保證 |
| **Steady-state reload 1.3 ms** | 同上 | 同上；另注意 reload 量測的是「per-tick wall time」不是「per-file parse time」 |
| **5-anchor e2e fire-through baseline** | `docs/benchmarks.md` §4 | n=30 + bootstrap 95% CI 的 honest 統計處理，但 docker-compose 6-service stack 的 fire-through 不等於 production K8s 的 fire-through — 缺 cri-o overhead / kube-proxy iptables hop / etc |

### 2.2 Soak finding（[#459](https://github.com/vencil/Dynamic-Alerting-Integrations/issues/459) 已立案）

Soak 1000-tenant Run A (GOGC=20 stress) + Run B (GOGC=100 production-shape) 都觀察到 `sys_bytes` / `heap_idle` high-water creep（+52% over 30min）。**決策**：bounded（39 MiB ≪ 100-500 MiB pod limit），歸因為 Go runtime scavenger 行為。

**Reviewer 該追問**：(a) 30 分鐘量測是否足夠代表 24h+ 生產壽命？(b) `+52%` 數字若 plot extrapolation 到 24h 會撞 pod limit 嗎？(c) GOGC=20 vs GOGC=100 兩種 runtime tune 差異收斂在哪裡？— 全部留到 v2.9.0 [#459](https://github.com/vencil/Dynamic-Alerting-Integrations/issues/459) 處理；本 release 接受 30-min 量測作為 confidence-level baseline。

### 2.3 4-hr soak deferred 到 [#459](https://github.com/vencil/Dynamic-Alerting-Integrations/issues/459)

v2.8.0 不執行真實 4-hr soak。**決策依據**：1000-tenant 30-min Run A+B 已給足 confidence；4-hr wall-clock 對 v2.8.0 ship timeline 影響大於收益（[#459](https://github.com/vencil/Dynamic-Alerting-Integrations/issues/459) 已 codify 為 v2.9.0 follow-up）。Reviewer 若不認可此決策應在 release approval 階段提出。

## 3. 大影響的「看起來小」變更

### 3.1 bump_docs sweep ([#439](https://github.com/vencil/Dynamic-Alerting-Integrations/pull/439))

191 files 機械 v2.7.0 → v2.8.0 跨四條 release line。**看起來像 mechanical no-op，但有 3 個 silently-introduced 修正**：

- Root `.md` frontmatter（CHANGELOG / README / README.en / CLAUDE）漏掉的 bump（bump_docs glob 不到 root，靠 Windows-vs-Linux symlink trap #39 才暴露）
- Generator hardcoded `version: v2.7.0` 在 `generate_doc_map.py` / `generate_tool_map.py` 各 2 sites（每次 regen 會回寫 v2.7.0，無限 drift loop）
- CHANGELOG L685 historical reference `登記於 v2.7.0` 被 bump_docs 誤 flip → 改 backtick wrap 才能保留歷史正確性

**Reviewer 該問**：同類 historical-vs-current 模糊還有沒有其他漏網？

### 3.2 codename leak sweep ([#468](https://github.com/vencil/Dynamic-Alerting-Integrations/pull/468))

22 customer-facing files / ~213 codename → feature-name 替換。Lint 加 `DEC-X` / `version-suffix` 兩 pattern + `PER_FILE_ALLOWLIST` 新機制收 ADR-019。

**Reviewer 該問**：(a) 22 個檔案的 prose rewrite 有沒有 introduce factual error（codename 換成 feature name 時若記錯哪個 codename 對應哪個 feature → silent 文意 drift）。例如 `B-3` 在不同上下文可指 Tenant API basic OR WatchLoop debounce metrics。(b) PR template 加的 codename checklist 是否會 churn 後續 PR author 體驗。

### 3.3 PR template 升級

[#468](https://github.com/vencil/Dynamic-Alerting-Integrations/pull/468) 用 specific codename catalog 取代「No codenames or jargon」模糊 line。**設計理由**：author-time discipline 不夠時，至少把「哪些 specific shape 不能寫」inline 列出，配合 lint mechanical net。Reviewer 若覺得太細可以 push back。

## 4. 可能 debatable 的設計選擇

| 選擇 | Why | Counter-arg reviewer 可能會提 |
|---|---|---|
| Phase 1 SSOT pilot 工具保留 dormant（不執行 ZH→EN 全量遷移） | premise validation 失敗（open-source SSOT 該 EN 無 contributor pool 佐證；3 條 trigger 條件 codify） | "已經寫了 pilot 工具，沉沒成本太大，先做了再說" — 反駁見 `ssot-language-evaluation.md` §0 superseded note |
| `check_codename_leak` 只走 Layer 1 enumeration（不在 v2.8.0 上 Layer 2 glossary gate） | Layer 2 需重整 glossary.md 結構 + 寫新 lint，scope 大；ship blocker 較少；[#469](https://github.com/vencil/Dynamic-Alerting-Integrations/issues/469) 已 codify follow-up | "永遠 lag 的 mechanism 不該 ship 為 final solution" — 反駁：Layer 1 是 historical 共業清掃，Layer 2 是 future self-healing；兩階段必要 |
| Migration Toolkit 三條交付路徑全 ship vs 只 ship Docker | air-gapped (金融/政府/軍工) 客戶需求明確 codify | "你還沒有實際 air-gapped 客戶，是 speculative 開發" — 反駁：Layer 1 cosign / SBOM / static binary cross-compile 是 supply-chain 通用 hygiene，Layer 2/3 (GPG / HSM / FIPS / SLSA L2-3) 才是 customer-RFP-driven |
| `da-batchpr` JSON-input-first contract（convenience flags deferred） | machine-friendly + automation-friendly；v1 contract 鎖住 | "customer 第一次跑會嫌不友好" — 反駁：customer-facing 是 da-tools Python 包裝，convenience flags 那層；Go binary 是 plumbing |

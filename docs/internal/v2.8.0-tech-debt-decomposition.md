## 1. 估點尺度（Scale Reference）

| 尺度 | 有效工時 | 典型特徵 | 日曆時間（含 CI + review） |
|---|---|---|---|
| **XS** | 0.5 天 | 一行改動 / 單 attr 加入 / 單 token 值替換 | 0.5-1 天 |
| **S** | 1-2 天 | 單檔多處 / 中等 regression 面（需 axe + 視覺 review 雙驗證） | 2-3 天 |
| **M** | 3-5 天 | 跨檔 / token 定義層變更（需 light/dark 雙模式 regression） | 1 週 |
| **L** | 1-2 週 | 設計層需決策 / 跨 component 重構 / 需 design partner 介入 | 2-3 週 |

> 有效工時以 §1.5 產能假設（DEC-K 草案）的 **30h/週 × 0.7 context-switch 稅** 為基數。L 尺度項目超標時**強制降規**或拆 Phase。

---

## 2. 分類總覽（性質分組）

9 項 TECH-DEBT 可分為 **兩大類三批次**：

### 2.1 Token 定義層（design-tokens.css 層）

根源在 `docs/assets/design-tokens.css` 的色值定義對比不達 AA。修一處影響全部使用該 token 的工具，**不能逐工具修**（會產生 token/JSX 雙軌）。v2.7.0 Day 7 已用相同模式修過 TECH-DEBT-003/-004，可複用經驗。

涵蓋：TECH-DEBT-005 / -007 / -010。

### 2.2 JSX Accessibility One-liners

JSX 作者層級慣例缺失（缺 `aria-label` / `tabindex` / `<label>`）。每項修復皆為一行，但**分散在 4 個工具**，合理化為 **per-tool PR** 或一次性 batch PR。

涵蓋：TECH-DEBT-006 / -008 / -009 / -011 / -012。

### 2.3 Token Migration 附帶項（需隨 A-3/A-4 遷移）

非獨立工作，會在 playground 納入 token 遷移時**自動消除**（前提：TECH-DEBT-007/-010 的 token 定義層已修正）。

涵蓋：TECH-DEBT-013。

**有效工時加總**：
- Token 定義層：0.5 + 1.5 + 1.5 = **3.5 天**
- JSX a11y one-liners：0.25 × 5 = **1.25 天**
- Migration 附帶：1 天
- **總計 5.75 天有效工時 ≈ 1.2 有效工時週**

套用 §1.5 產能假設 30h/週 × 0.7 稅（Phase .a 與 Phase .b 平行期）：
**日曆時間 ≈ 5.75 / (30 × 0.7 / 8) = 5.75 / 2.625 = ~2.2 日曆週**
（計算說明：30h × 0.7 = 21h 有效/週；21h / 8h/日 = 2.625 有效日/日曆週；5.75 有效日 / 2.625 = 2.19 日曆週）

## 4. 批次施工策略

為避免 9 個獨立 PR 造成 review fatigue，建議**批次 3 PR**：

### PR #1 — Token Audit Batch（M 規模）

**範圍**：TECH-DEBT-005 + TECH-DEBT-007 + TECH-DEBT-010
**改動點**：`docs/assets/design-tokens.css` + 可能的 `threshold-heatmap.jsx` / `multi-tenant-comparison.jsx` / `cost-estimator.jsx` 局部重用新 token
**驗收**：
- `tests/e2e/_axe-audit-day1to3.spec.ts` + `_axe-audit-day4.spec.ts` 全綠
- Light + Dark 雙模式手動視覺 review（ADR-016 單軌規範）
- `make scan-component-health` 相關 tool 回到 `healthy` Tier
**預估**：3.5 天有效工時 + 2 天 review/CI/視覺驗證 = **~5.5 工作日 ≈ 8 日曆日**
**依賴**：TECH-DEBT-010 設計決策（方案 A vs B）需在 PR 開工前拍板
**Owner**：建議單人連做（避免 token 合併衝突）

### PR #2 — JSX a11y Sweep（S 規模）

**範圍**：TECH-DEBT-006 + -008 + -009 + -011 + -012（5 項一行 a11y 修復）
**改動點**：
- `threshold-heatmap.jsx`（-006）
- `multi-tenant-comparison.jsx`（-008）
- `notification-previewer.jsx`（-009）
- `playground.jsx`（-011, -012 同檔合修）
**驗收**：
- 4 個工具的 axe runtime spec 全綠
- 新增 ESLint rule（提案）：`jsx-a11y/scrollable-region-focusable`（已存在 [jsx-a11y plugin](https://www.npmjs.com/package/eslint-plugin-jsx-a11y)），若未啟用則於此 PR 打開
- `dev-rules.md` §互動工具變更 SOP 補「所有 `role="region"` + scrollable 容器必附 `tabindex={0}`」段落
**預估**：1.25 天有效工時 + 1 天 review/CI = **~2.5 工作日**
**依賴**：無（純 JSX 層）
**Owner**：可與 PR #1 平行，不同 reviewer

### PR #3 — Playground Token Migration Tail（S 規模）

**範圍**：TECH-DEBT-013
**改動點**：`playground.jsx` 殘留 Tailwind palette → design token
**前置**：PR #1 先 merge（-007/-010 token 定義更新後才能套）；A-3/A-4 的 playground 遷移批次若已涵蓋 playground 則併入 A-3/A-4
**驗收**：playground 的 axe color-contrast 完全消失（x3 → 0）
**預估**：1 天有效工時 + 0.5 天 CI = **~1.5 工作日**
**Owner**：可延到 Phase .a 後半（A-3/A-4 收束時）

**結論**：若嚴格套 §1.5 產能假設（30h × 0.7 稅），**Phase .a 實務日曆時程應為 11-14 週，而非原估 4-5 週**。需 R0 決策：

1. **砍範圍**（建議）：軌道二 A-12~A-16 部分延 v2.8.1（A-12/A-13 留本版，A-14/A-15/A-16 延），Phase .a 收斂到 6-8 週

3. **提高產能假設**：DEC-K 調高週有效工時（30h → 40h+）或降低 context-switch 稅（0.7 → 0.85），重做數字

**推薦**：方案 1 + 方案 3 組合 — 把 A-12/A-13（環境攔截 + Playwright lint）**必做**，A-14/A-15/A-16 降為 **nice-to-have**；同時 DEC-K 針對 Phase .a 單任務期調高 context-switch 係數到 0.85（理由：軌道一多為獨立 PR，context-switch 成本低於平行 .b benchmark 期）。

## 6. 風險

1. **TECH-DEBT-010 設計決策卡住**：方案 A vs B 需設計 + dev 討論，若決策拖過 PR #1 開工日 → 把 -010 拆出 PR #1 獨立 PR，-005/-007 先走。
2. **PR #2 JSX sweep 併 PR conflict**：5 個檔案同時改動，若其他 A-3/A-4 遷移 PR 同時動到 playground.jsx，須 rebase。緩解：PR #2 優先、早 merge。
3. **TECH-DEBT-013 自動消除比例不明**：-007/-010 token 定義修正後，playground 3 處 contrast 可能部分自動達 AA 但不保證全部。PR #3 開工前先跑一次 local axe re-scan 確認殘留數。
4. **ESLint jsx-a11y plugin 擴 rule 引入其他 regression**：開啟 `scrollable-region-focusable` 可能暴露 PR #2 範圍外的其他工具殘留問題 → 擴大 scope。緩解：先 lint 跑 `--no-error-on-unmatched-pattern`，只 warn 不 fail 一週，累積待辦。

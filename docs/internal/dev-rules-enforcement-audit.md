# Dev-Rules Enforcement Audit (v2.8.0)

> **Purpose**: Cross-reference each rule in [`dev-rules.md`](dev-rules.md) against the actual pre-commit hook / CI gate that enforces it. Surfaces three categories: (a) **enforced** (rule has working code-driven gate), (b) **doc-drift** (rule claims a hook that doesn't exist), (c) **doc-only** (rule explicitly relies on agent reading + reviewer judgment, no hook expected).
>
> **Why now**: PR #166 (`check_subprocess_timeout.py` for S#74 Code-driven Layer A) demonstrated the value of converting agent-discipline lessons into machine-checkable lints. Same approach applied to dev-rules surfaces 4 doc-drift gaps and 3 high-ROI codification candidates.
>
> **Generated by**: manual audit (PR #168, 2026-05-01). Future regen: any time a rule's enforcement claim changes.

## Recommendations (ranked by ROI)

### 🟢 Tier A — high-ROI fixes (recommend ship within v2.8.0)

**A-1. Fix Rule #2 Tenant-Agnostic doc-drift**

Two paths:
- **Quick**: edit dev-rules.md to remove the `lint_hardcode_tenant` claim, mark as "process rule, reviewer enforced"
- **Better**: write the missing lint. The rule pattern is clear: scan Go / PromQL / Rule Pack YAML for hardcoded tenant ids matching a known list (`db-a`, `db-b`, etc) outside of fixture / test allowlists.

**Estimated**: quick fix = 5 min; real lint = ~80 LOC + 20 tests.

**A-2. Fix Rule #5 SAST 7 rules doc-drift**

The 7 rules listed (`encoding/shell-safe/chmod/yaml.safe_load/credentials/dangerous-fns/stderr-routing`) are partially covered by external tools (bandit, safety) but **not as pre-commit hooks in this repo**. Two paths:

- **Quick**: doc fix — rename section to "SAST guidelines" (recommendations) instead of "auto scanning rules"
- **Better**: add `bandit` as a pre-commit hook with the 7-rule profile. Already a Python ecosystem standard; ~10 LOC of `.pre-commit-config.yaml`.

**A-3. Fix Rule #6 推銷語言 doc-drift**

`check_marketing_language` hook claimed but not found. Two paths:
- **Quick**: doc fix — remove claim, mark as "reviewer convention"
- **Better**: write the lint. ~50 LOC scanning `README*.md`, `docs/**/*.md`, commit messages for keyword list (`業界領先 / 革命性 / 唯一 / 世界第一` etc).

### 🟡 Tier B — extensions to existing partial enforcement

**B-1. Extend Rule #4 doc-as-code** — currently `doc-map-check` validates `doc-map.md` is up-to-date, but doesn't catch the case "API change in code without CHANGELOG entry". Could be a CI gate that triggers when `components/**/*.go` or `scripts/tools/**/*.py` changes without a CHANGELOG diff in the same PR.

**B-2. Extend Style S1 (no slate)** — `design-token-usage` hook exists (manual stage) but doesn't specifically forbid `slate`. Add as a banned-token list.

**B-3. Extend Style S5 (semantic token brightness)** — design-system-guide.md has the rule. Could compute brightness deltas of token usages and lint when exceeded. Heavier (~150 LOC).

### ⚫ Tier C — explicitly process-only (no action)

Rules 1, 3, 8, P3, A1, A4, A5, A7 are by-design process / design-pattern rules with no expectation of code enforcement. Doc explicitly should call this out so future agents don't waste time looking for the hook.

## How to maintain

When adding a new rule to `dev-rules.md`:

1. **Decide enforcement category up front** — `enforced` / `partially enforced` / `doc-only by design` (NOT `aspirational`).
2. **If enforced**: add the lint script + pre-commit hook in the same PR.
3. **If partially enforced**: explicitly note the gap.
4. **If doc-only**: do not write "**檢查方式**: `pre-commit hook foo`" — say "**reviewer convention**" or "**process rule**".
5. **Update this audit doc** in the same PR (add row to relevant table).

The rule "if the lint claim makes it through review, the lint must exist" is itself a process rule that this audit codifies.

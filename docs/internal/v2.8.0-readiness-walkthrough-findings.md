## §1. B-1 Install runbook — 文件 vs 實際 artifacts 稽核

照客戶角度從 `docs/getting-started/for-platform-engineers.md` + `docs/migration-toolkit-installation.md` 從上跑到下，每個 `kubectl` / `make` / `python3` / `da-tools` 指令都對 repo 實際 artifacts 比對。發現 **6 處 customer-blocking 落差**，全數 patch 在本 PR 內。

### Gap 1 — `kubectl apply -f k8s/02-threshold-exporter/` 路徑不存在

| 項目 | 內容 |

| 出處 | `for-platform-engineers.{md,en.md}` §部署 threshold-exporter |

## §2. B-3 Compressed-time chaos + drift harness

### §2.1 設計

替代「4 小時實機 wall-clock soak」的工程化版本。harness 兩支腳本 + 兩個 Makefile target：

```
scripts/tools/dx/run_chaos_soak.py   ← 主 orchestrator
scripts/tools/dx/render_soak_diff.py ← 結果 renderer
Makefile target soak-readiness        ← 4hr default
Makefile target soak-readiness-smoke  ← 2min CI-friendly
```

**為什麼不用 pprof**: threshold-exporter / tenant-api 預設不 expose `net/http/pprof`（暴露程式結構 + perf surface 屬獨立 hardening 議題，不在 v2.8.0 scope）。改用 Prometheus `/metrics` 端點上既有的 `go_*` runtime collector — 對「是否 leak」的問題訊息一致，且不需動 production code。

**為什麼不用 `process_*` collectors**: threshold-exporter 用 promauto 預設只註冊 Go runtime collector，**沒**註冊 `prometheus.NewProcessCollector`。`go_memstats_sys_bytes`（從 OS 取得的總 bytes）是 Go program 的 RSS proxy，等價訊號。

**Tracked metrics**:
- `go_goroutines`（goroutine 洩漏偵測 — 最重要的單一訊號）
- `go_memstats_sys_bytes`（RSS proxy）
- `go_memstats_alloc_bytes`、`heap_inuse_bytes`、`heap_idle_bytes`、`heap_objects`（heap 分層）
- `go_gc_duration_seconds_count`（GC 活動 — informational only）

**Drift verdict**:
- PASS：drift < 5%
- WARN：5% ≤ drift < 20%
- FAIL：drift ≥ 20%
- 預設跳過前 10% samples 作 warmup（30s-5min clamp）— 解決 Go cold-start arena 分配在前 30-60s 的偽信號

### §2.2 短版 smoke 驗證結果

dev container 內跑 `threshold-exporter` 於 `:8080`，conf.d fixture 兩份 tenant：

| Run | 11 reloads / 23 polls | duration 3 min |
|---|---|---|
| Metric | First → Last (drift) | Verdict |
| `go_goroutines` | 9 → 9 (+0.0%) | PASS |
| `go_memstats_sys_bytes` | 17.6 MiB → 17.6 MiB (+0.0%) | PASS |
| `go_memstats_alloc_bytes` | 0.8 MiB → 0.8 MiB (-0.2%) | PASS |
| `go_memstats_heap_inuse_bytes` | 2.6 MiB → 2.8 MiB (+7.9%) | WARN |
| `go_memstats_heap_idle_bytes` | 8.8 MiB → 8.5 MiB (-3.0%) | PASS |
| `go_memstats_heap_objects` | 3198 → 3131 (-2.1%) | PASS |

**結論**: harness 工作正常 — 識別出 1 個 WARN（heap_inuse +7.9%），其餘全 PASS。3-min run 的 +7.9% 在 4-hr run 應收斂；尤其 `go_memstats_sys_bytes`（最重要的 RSS proxy）+ `go_goroutines` 都 +0.0%，沒有結構性 leak signal。

### §2.3 4-hr 真實 soak 該怎麼跑

```bash
# Terminal 1 — 啟 threshold-exporter
go build ./components/threshold-exporter/app
./threshold-exporter --config-dir <your conf.d> --listen :8080 &

# Terminal 2 — 跑 4-hr soak
make soak-readiness

# 結果
# .build/v2.8.0-soak/metrics-timeseries.csv   每 30s 一筆 sample
# .build/v2.8.0-soak/soak-report.md           人讀 report
# .build/v2.8.0-soak/run-config.json          run 參數 + 統計
```

預期：4-hr / ~240 reloads / ~480 metric polls。`make soak-readiness-smoke` 給 CI 驗證 harness 還活著（不替代真實 soak）。

- 🟡 4-hr 真實 soak 尚未實跑 — harness 已 ready，待實機環境執行

- 🟡 Air-gapped tar import path（`Path C`）— 文件 patch 過了，但實際 tar 載入流程沒有獨立驗證

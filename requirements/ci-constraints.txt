# CI / test Python dependency constraints — SSOT (#1158 root-solve).
#
# HOW THIS IS USED
#   Every `pip install <pkgs>` step in .github/workflows/** and the
#   devcontainer references this file with `-c requirements/ci-constraints.txt`.
#   As a pip *constraints* file it does NOT install anything on its own —
#   it only pins the version of a package IF that package is being
#   installed. So each job keeps its own readable, explicit dependency
#   list; only the versions are centralized here.
#
# WHY PINNED
#   `pip install <pkg>` with no version tracks *latest* on every CI run.
#   For behavior-affecting tools (test runner / generator / coverage /
#   schema validation) a new release can silently drift test outcomes,
#   rename a health check, or flip a coverage gate — the "local green,
#   CI first-run red" version-drift class (#1158, originally surfaced by
#   the schemathesis `filter_too_much` seed-flake). Freezing every CI
#   Python dep here makes runs reproducible and turns each upgrade into
#   an explicit, reviewable PR (Renovate opens them — see renovate.json).
#
# UPGRADE
#   Do NOT hand-edit to chase latest. Let Renovate bump a line via PR, or
#   bump deliberately and re-verify. A new unpinned `pip install` is
#   blocked by the `check_unpinned_deps` gate (scripts/tools/lint/).
#
# Versions below = the latest at pin time = what unpinned installs were
# already pulling, so pinning is behavior-preserving.
